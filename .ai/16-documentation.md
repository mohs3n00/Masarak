# Documentation Rules

Documentation is part of the product.

Undocumented code is considered incomplete.

---

# General Principles

Documentation must be:

Accurate

Current

Simple

Useful

Maintainable

---

# Code Documentation

Document:

Complex logic

Business rules

Algorithms

Architecture decisions

Avoid documenting obvious code.

---

# Function Documentation

Document when necessary:

Purpose

Inputs

Outputs

Exceptions

Side effects

---

# Component Documentation

Document:

Purpose

Props

Dependencies

Usage

Limitations

---

# API Documentation

Every API should document:

Purpose

Authentication

Permissions

Parameters

Responses

Errors

Examples

---

# Database Documentation

Document:

Tables

Relationships

Indexes

Constraints

Migrations

Business rules

---

# Business Rules

Document important business logic.

Future developers should understand WHY the rule exists.

---

# Configuration

Document:

Environment variables

Required services

External dependencies

Secrets (names only, never values)

---

# Installation

Document:

Requirements

Installation

Configuration

Running locally

Testing

Deployment

---

# Project Structure

Document:

Main folders

Architecture

Modules

Shared components

Responsibilities

---

# Changelog

Document major changes.

Document breaking changes.

Document migrations.

---

# README

Every project should explain:

Purpose

Features

Technology stack

Setup

Development

Testing

Deployment

License

---

# Examples

Provide examples whenever possible.

Examples are better than long explanations.

---

# Keep Documentation Updated

Whenever code changes:

Review related documentation.

Update if necessary.

Never leave outdated documentation.

---

# Documentation Checklist

✓ APIs documented

✓ Complex logic documented

✓ Architecture documented

✓ Configuration documented

✓ README updated

✓ Examples included

✓ No outdated information
